Wells Coding Harness

A local, model-agnostic agentic coding harness. Give it a development goal and a crew of AI agents actually reads files, makes edits, runs tests, and ships a PR — Claude-Code style, driven by any model you choose.

★Current Capabilities (July 2026)

Beyond the core loop described below, Wells now ships:

  • Full-screen TUI — multi-line prompt with persistent history, live status bar (tokens + dollar cost + agent activity), Escape-to-cancel, streamed answers, modal /config settings and /mcp server manager.
  • Deterministic verification — the harness runs the real test suite as a gate (fail-fast back to the coder), lint/syntax self-heal after every edit, and whitespace-tolerant edits with live colorized diffs.
  • Structural repo index (wells-index, Rust/tree-sitter, on PyPI) — instant symbol lookups plus a goal-ranked repo map injected into planner/coder prompts.
  • Git safety net — every run snapshots the working tree; /undo reverts a run; per-node session checkpoints make /resume crash-proof; opt-in auto-commits.
  • MCP both directions — Wells is an MCP server for other agents and an MCP client that pulls external tools (databases, docs, GitHub) into the agent's toolset, all behind the safety gate.
  • Operating modes — /mode plan | approve | auto | dryrun, parallel read-only research fan-out, per-run token budgets, cheap-model routing for verification.

One goal in, a verified code change out. Six reasoning stages run in a loop, with the middle three being autonomous tool-using agents that operate on your real workspace.

What makes this Claude-like: The Coder, Tester, and Reviewer don't just describe what they would do — they call real tools (read_file, edit_file, run_tests, …) in a model-driven loop, observe the results, and iterate until the work is genuinely done. The Reviewer independently re-checks by reading the changed files and re-running tests before signing off.

This is where the agent actually works. Each of the Coder / Tester / Reviewer nodes spawns an executor that runs this loop with its own task and toolset.

Native tool-calling

Models that support OpenAI/Anthropic-style tool_calls (Z.ai GLM, GPT, Claude, Gemini…) bind the tool schemas directly. Fastest, most reliable path.

auto-detected

Text fallback

Models without native tool support emit <tool_call>{json}</tool_call> blocks in their text. The harness parses and dispatches them — so any decent AI can drive it.

universal compatibility

Subagent spawning

The model can call spawn_subagent to delegate a focused research (read-only) or fix task — keeping the parent's context lean. Runs in parallel via threads.

token-efficient parallelism

4The Tool Layer

Eight tools cover the full edit-test-verify cycle. Reads are always allowed; writes and shell commands pass through the safety gate.

read_file list_dir glob grep write_file edit_file run_command run_tests spawn_subagent

green = read-only, always allowed  ·  orange = mutating, safety-gated  ·  teal = delegation

Safety policy (HARNESS_SAFETY)

ModeBehaviour
autoExecute immediately, confined to WORKSPACE_ROOT. Destructive commands (rm -rf /, mkfs…) are always blocked.
approveRequire an approval callback. Degrades to dry-run when no callback is wired, so nothing destructive happens unattended.
dryrunNever execute — tools describe what they would do. Truly side-effect free.

PLAN_MODE=1 forces all mutating tools to simulate, so you can preview exactly what the agent would change before allowing it.

Categorized chunks, stable-prefix ordering for prompt-cache hits, priority-based budget trimming when over budget.

Log compressor

Strips ANSI, collapses duplicates/blanks, keeps a tail window, preserves tracebacks. Tool outputs auto-compressed before going back to the model.

On loop iterations, verbatim plan/architecture is replaced with a summary once durable context exceeds SUMMARIZE_THRESHOLD.

Model router

Low-stakes tasks (summarization, compression, classification) route to MODEL_PROFILE_CHEAP when configured.

Cache-friendly prefix

SystemMessage + deterministic chunk order maximizes provider-side prompt-cache hits — visible as cache_read savings in the report.

7MCP Server — talk to it from anywhere

The harness runs as a Model Context Protocol server over stdio, so external agent clients (Claude Code, OpenCode, Codex CLIs, Gemini CLI…) can invoke it.

run_agent_task plan_task review_code run_executor spawn_subagent search_repo read_file run_command git_status get_memory compress_logs get_harness_info

Start it with coding-harness-mcp or python -m coding_harness. The MCP layer is deliberately thin — it delegates every call to the harness core, no duplicated logic.

Inside the REPL, you can type your goals naturally or use slash commands (/help, /config, /info, /plan, /quit).
