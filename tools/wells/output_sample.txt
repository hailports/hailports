ARCHITECTURE PROPOSAL
======================================================================
Here is a concrete architecture and design proposal based on your development plan. Because Phase 1 (Discovery) is not yet complete, this proposal uses a highly adaptable, framework-agnostic front-end architecture that can be easily implemented in React, Vue, or Vanilla JS.

### 1. Components / Modules and Their Responsibilities

To maintain separation of concerns and keep the code testable, the feature should be split into three distinct logical modules:

*   **`PageTreeBuilder` (Pure Logic Module)**
    *   *Responsibility:* Takes the raw, flat array of page URLs/objects and transforms it into a nested hierarchical tree structure based on the `/` delimiters. Contains zero DOM manipulation logic.
*   **`PageTreeFolderView` & `PageTreeItemView` (UI Components)**
    *   *Responsibility:* Renders the nested tree.
        *   *FolderView:* Renders an accordion header with an expand/collapse toggle (e.g., using native `<details>/<summary>` tags or simple state toggles for maximum simplicity and native a11y). Contains a list of children.
        *   *ItemView:* Renders the selectable page leaf. Emits a `selectPage(pageId)` event when clicked.
*   **`PagePickerContainer` (Orchestrator Component)**
    *   *Responsibility:* Replaces the old multiline textbox wrapper. Holds the state for `expandedFolders` and `selectedPageId`. Renders the TreeView components. Hosts the new **"Load"** button, managing its disabled state based on whether `selectedPageId` is populated.
    *   *Editor Integration:* Contains the bridge function that calls the Live Preview Editor's load API.

### 2. Key Data Models / Schemas

The most critical part of this build is the data transform. We need to shift from a flat list to a recursive tree.

**Input Schema (Assumed Flat List)**
Assuming the existing textbox was populated by an array of strings or simple objects:
```typescript
interface FlatPage {
  id: string;       // unique identifier
  url: string;      // e.g., "/about/team"
  title?: string;   // e.g., "Our Team"
}
```

**Output Schema (Nested Tree)**
The `PageTreeBuilder` will output recursive nodes representing either a folder or a page:
```typescript
type TreeNode = TreeFolder | TreePage;

interface TreeFolder {
  id: string;          // e.g., "folder-about"
  name: string;        // e.g., "about"
  path: string;        // e.g., "/about"
  type: 'folder';
  children: TreeNode[];
}

interface TreePage {
  id: string;          // original page id
  name: string;        // e.g., "team" (derived from URL)
  path: string;        // e.g., "/about/team"
  type: 'page';
  children?: never;    // leaves have no children
}
```

### 3. External Dependencies and Libraries

*   **No external tree libraries recommended.** To adhere to *Simplicity First*, avoid heavy dependencies like `react-arborist` or `jqtree`.
*   **HTML Native Accordion (Recommended):** If the project allows, use semantic HTML `<details>` and `<summary>` elements for the folders. This provides the accordion show/hide behavior, keyboard accessibility, and `aria-expanded` states natively with zero JavaScript overhead.
*   *(If the project already uses a UI framework like Tailwind, Material, or Bootstrap, utilize their built-in Accordion/List components to match existing project style).*

### 4. Risk Areas / Trade-offs

*   **Performance Risk (DOM Bloat):** If the system has thousands of pages, rendering the entire expanded tree at once will freeze the browser.
    *   *Trade-off/Mitigation:* The design uses "collapsed by default" folders (Phase 2 requirement). If performance is still an issue during verification (Phase 5), we will need to introduce "lazy loading" (only render children when a folder is expanded) or windowing/virtualization. *Out of scope unless observed.*
*   **Editor Integration Unknowns (Phase 1 Dependency):** The "Load" button is entirely dependent on the Live Preview Editor's API. If the editor requires a full page refresh, an iframe `src` update, or a specific postMessage format, the `PagePickerContainer` will need to adapt.
    *   *Trade-off/Mitigation:* Isolate the editor trigger in a single function `loadIntoEditor(pageId)` inside the orchestrator. If the API is complex, this function will be the only thing that needs to change.
*   **URL Parsing Edge Cases:** Pages with trailing slashes (`/about/`), query parameters (`/about?lang=en`), or lack of URL encoding could break a naive string-split tree builder.
    *   *Mitigation:* The `PageTreeBuilder` must sanitize inputs, strip trailing slashes, and group pages cleanly before generating the `TreeFolder` nodes. Empty folders (if any) should be safely ignored or explicitly rendered depending on UX preference.
